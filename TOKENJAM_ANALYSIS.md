## 🔥 ГОРЯЧИЕ ТОЧКИ (что тратит токены)

### 1️⃣ **Read_file — 4.4 МБ (85% всех инструментов)**
- **915 вызовов**, в среднем 4,934 байт на вызов
- **Проблема:** Агент читает одни файлы повторно (нет кэша)
- **Пример:** Один файл читается 5-10 раз за сессию
- **Экономия:** 30-40% сокращение с LRU-кэшем

### 2️⃣ **Grep/Search — 800 вызовов (210 КБ)**
- **590 × grep_files** + 94 × search_in_file + 7 × grep_codeviper_files
- **Проблема:** Агент ищет один паттерн несколько раз
- **Экономия:** 20% с кэшем результатов grep за сессию

### 3️⃣ **Run_command — 348 вызовов (371 КБ)**
- Достаточно выходов, но много дублей
- **Пример:** `npx vitest run` запускается 15 раз

### 4️⃣ **Системный промпт + инструменты**
- **Актуальный размер:** ~5-8 КБ на запрос
- **Проблема:** Все инструменты описаны в JSON каждый раз
- **Экономия:** 15-20% с выключением неиспользуемых инструментов в нужных режимах (уже частично реализовано)

### 5️⃣ **История контекста**
- При 30-50 сообщениях в чате: +20-30% к размеру контекста
- **Проблема:** Старые инструменты tool_results остаются в контексте
- **Экономия:** 10-15% с обрезкой tool результатов (п. 6 ROADMAP)

## 💡 РЕКОМЕНДАЦИИ (приоритет по экономии)

### 🥇 **1. LRU-кэш для `read_file` / `read_codeviper_file` (25-35% экономия)**
- **Файл:** `app/electron/main/fileSearchInWorker.ts` (п. 3 ROADMAP)
- **Как:** Кэш 500 записей по ключу `{path, offset?, limit?}`
- **Инвалидация:** По `mtime` файла
- **Стоимость:** 2-3 часа
- **Пример:**
  ```typescript
  const cache = new Map<string, {content: string, mtime: number}>()
  // Кэш существует для embeddings, переиспользовать логику
  ```
- **Экономия в токенах:**
  - read_file: 4,408 КБ → ~2,600 КБ (40% сэкономится)
  - Каждый прочитанный файл → в контекст LLM один раз
  - **В целом: ~200-250k токенов в день**

### 🥈 **2. Батчинг grep/search запросов (15-20% экономия)**
- **Файл:** `app/electron/main/fileSearch.ts`
- **Как:** Группировать параллельные grep запросы в worker
- **Текущее:** 1 grep = 1 worker call
- **Новое:** Объединить до 4-5 патернов в один batch
- **Стоимость:** 1-2 часа
- **Экономия:** grep выводы: 210 КБ → ~170 КБ (~100-150k токенов)

### 🥉 **3. Настраиваемый порог суммаризации (10-15% экономия)**
- **Файл:** `app/electron/main/agentContext.ts`
- **ROADMAP:** п. 4, 14
- **Как:** Позволить пользователю выбрать агрессивное сжатие (65% вместо 85%)
- **Стоимость:** 1 час + UI
- **Экономия:** На длинных сессиях (>50 сообщений)
  - **Текущее:** 85% → суммаризация при 14k токенов
  - **Агрессивное:** 65% → суммаризация при 8.5k токенов
  - **Экономия:** ~15-20% на средней сессии (~50-100k токенов)

### 💾 **4. Автообрезка вложенных файлов (5-10% экономия)**
- **ROADMAP:** п. 7
- **Как:** Файлы > 20 КБ обрезать (первые + последние 50 строк)
- **Стоимость:** 1.5 часа
- **Текущее:** Если файл 100 КБ → весь в контекст
- **Новое:** Файл 100 КБ → 5 КБ в контекст
- **Экономия:** 50-100k токенов на сессию с большими файлами

### 🧹 **5. Удаление старых tool_result из контекста (10-15% экономия)**
- **ROADMAP:** п. 6
- **Как:** Оставлять результаты только последних 5 вызовов
- **Стоимость:** 1 час
- **Текущее:** Все tool_result идут в контекст
- **Пример:**
  ```
  История: read file 1, grep, grep, grep, read file 2, grep, grep
  Новое: показать только последние 5: читав file 2 + 4 последних grep
  ```
- **Экономия:** На длинных диалогах (30+ шагов): 50-100k токенов

### 📌 **6. Кэш результатов grep (5-8% экономия)**
- **Файл:** `app/electron/main/fileSearch.ts`
- **Как:** Кэш поиска по ключу `{query, path, glob}`
- **Инвалидация:** По `mtime` любого файла в path
- **Стоимость:** 1 час
- **Экономия:** 30-50k токенов на активных сессиях
