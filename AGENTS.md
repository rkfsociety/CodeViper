# Правила проекта CodeViper

**Память и правила для Cursor-агента:** [`.cursor/README.md`](.cursor/README.md) (правила в `.cursor/rules/`, уроки в `.cursor/README.md`). Этот файл — ROADMAP и самоулучшение; архитектура — `CLAUDE.md`.

## Сборка после изменений

- **После любого изменения кода в `app/` — всегда запускать `npm run build` в папке `app/` перед коммитом.** Пользователь не умеет и не должен собирать вручную — папка `out/` в коммите должна быть актуальной.
- Никогда не говорить пользователю «пересобери», «запусти build», «перекомпилируй» — это задача агента. Пользователь только запускает `CodeViper.cmd`.
- Порядок перед каждым коммитом: `npm run typecheck` → `npm run build` → обновить ROADMAP/README → `git commit` → `git push`.
- `npm run dist` автоматически вызывает `setup-node` (скачивание portable Node.js LTS в `app/resources/node/`). Папка в `.gitignore`, не коммитить.

## Версионирование

- **`version` — только при релизе оболочки** (см. `versioning.mdc`, **ROADMAP блок 0**).
- **Блок 0 (1–8) — первым**; пункты 9+ — после переноса блока 0 в «✅ Сделано».

## Portable Node.js

Цепочка самопересборки завершена: `setup-node` → `extraResources` → `getBundledNodeBin()` + PATH в `runCodeViperCommand`.

## После каждой выполненной задачи (ROADMAP)

- Удалить выполненный пункт из раздела «В планах» (или всю группу `### 🔗 …`, если в ней не осталось пунктов). **Выполненные пункты никогда не остаются в «В планах».**
- Добавить краткую запись в раздел «Сделано» в **ROADMAP_DONE.md** (объединять с похожими темами, без деталей реализации).
- **Перенумеровать все оставшиеся пункты «В планах» с 1** — без пропусков (после удаления пункта 5 бывший 6 становится 5).
- Обновить ссылку «1…N» в блоке «Формат задач для самообучения», если меняется общее количество пунктов.

## Коммиты после кода

- После любого изменения кода агент должен довести работу до `git commit` в этом же сеансе, а при возможности сразу и до `git push`.
- Не оставлять изменения кода без коммита, если задача уже выполнена и проверки пройдены.
- Исключение только одно: если пользователь явно просит не коммитить или остановить работу до коммита.
- Если пользователь присылает задачу из `ROADMAP.md`, агент обязан выполнить её end-to-end: прочитать пункт, внести кодовые изменения, прогнать проверки, перенести пункт в `ROADMAP_DONE.md`, обновить `ROADMAP.md` и завершить всё `git commit`-ом, а при возможности сразу `git push`-ом.

## Дорожная карта (ROADMAP.md)

### Формат каждого пункта «В планах»

Каждая задача — **атомарный пункт для одного прогона самообучения**, не таблица:

```text
N · [S/M/L/XL] · Краткое название — приор. High|Medium|Low
- Цель: один измеримый результат
- Файлы: конкретные пути (app/electron/main/…, app/src/…)
- Действие: одна атомарная правка
- Проверка: npm run typecheck; npm test -- … | сценарий в UI
```

**Промпт пользователя:** `Выполни пункт N из ROADMAP.md — самоулучшение CodeViper.`

### Структура раздела «В планах»

- Задачи **сгруппированы по теме** (`### 🔗 …` — цепочки; `### ⚡ Независимые задачи` — без жёсткого порядка между собой).
- **Сквозная нумерация** всех пунктов с **1** до конца раздела; группы **не сбрасывают** счётчик.
- Внутри **цепочки** (`🔗`) — порядок строго сверху вниз; следующий пункт только после предыдущего.
- **Независимые** пункты (`⚡`) — порядок произвольный.
- В `ROADMAP_DONE.md` номера **не используются**.
- Пустую категорию (заголовок без пунктов) — **удалить целиком**.

### Добавление новых задач в ROADMAP

- Новый пункт оформлять **только** по шаблону выше; в конец подходящей группы или новой цепочки.
- После добавления — **перенумеровать весь раздел «В планах» с 1**.
- Сложную идею **не писать одним абзацем** — разбить на несколько пунктов S/M, каждый с отдельной проверкой.

### Выполнение пункта агентом (самоулучшение)

1. `read_codeviper_file` → `ROADMAP.md`, найти пункт N (цель, файлы, действие, проверка).
2. Прочитать файлы из поля **Файлы**.
3. `set_self_improvement_plan` — шаги из **Действие** + **Проверка** (обычно 3–8 item'ов).
4. Правки → `run_codeviper_command` по **Проверке** → `complete_self_improvement_item`.
5. Перенести пункт в `ROADMAP_DONE.md`, перенумеровать `ROADMAP.md` — в том же коммите, что и код.

## Актуальность README

`README.md` — витрина проекта; обновлять **в том же коммите**, что и связанные изменения.

**Обновлять, когда:**
- новая пользовательская возможность (провайдер, интеграция, UI, P2P, MCP);
- изменился быстрый старт или требования;
- изменилось число пунктов ROADMAP (счётчик «N задач»);
- появился заметный сценарий — дополнить `docs/example-prompts.md` / `docs/demos.md`; в README — ссылка.

**Проверять:** краткий README, быстрый старт, ссылки на **вики** (основное) и `docs/` (GIF, промпты, интеграции), счётчик ROADMAP.

**Не делать:** длинные разделы в README; откладывать обновление `docs/` «на потом».

Подробный чеклист — `.cursor/rules/readme.mdc`.
