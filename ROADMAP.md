### Надёжность очередей загрузки и агента ✅

- [x] **38. `useOllamaDownloadQueue`: очередь останавливается при ошибке** — ошибка загрузки теперь помечает `succeeded = false` и продолжает цикл; модель всегда удаляется из очереди
- [x] **39. `useOllamaDownloadQueue`: гонка при проверке установленных моделей** — `enqueuedSetRef: Set<string>` отслеживает все добавленные модели сессии; `enqueue` проверяет Set вместо устаревшего `installedModels`
- [x] **40. `useMessageQueue`: гонка `stopAgent` ↔ `processNextQueuedRun`** — `stoppingRef` выставляется в начале `stopAgent`, сбрасывается в `finally`; `processNextQueuedRun` возвращается немедленно при `stoppingRef.current`
- [x] **41. `useMessageQueue`: при смене чата очередь не очищается полностью** — `resetQueue` инкрементирует `runIdRef` и синхронизирует `doneRunIdRef`; поступающие stream-события старого run игнорируются
- [x] **42. `useMessageQueue`: нет повтора при сбоях сети** — цикл `attempt 0..1`; сетевые ошибки (`fetch failed`, `ECONNREFUSED`) дают 1 автоповтор через 2 с с системным сообщением; остальные ошибки — сразу в очередь
- [x] **43. `useOllamaDownloadQueue`: нет таймаута на загрузку** — `Promise.race` с таймаутом 10 мин; при превышении — ошибка и переход к следующей попытке/модели
- [x] **44. `useOllamaDownloadQueue`: нет авто-повтора при сбоях** — до 3 повторов с задержкой 1 с → 3 с → 9 с; после всех попыток — `setError` и продолжение очереди

### P2P-сеть общих вычислений

Цель: пользователи объединяют мощности ПК — каждый отдаёт до 20% GPU/CPU в общий
пул и пользуется мощностями остальных по принципу «чем больше отдаёшь — тем больше
получаешь».

- [ ] **96. Сигнальный сервер** — VPS с реестром узлов (ID, GPU/RAM, нагрузка); Node.js + WebSocket + Redis
- [ ] **97. Аккаунтная система** — регистрация по email/GitHub OAuth; токен определяет репутацию и лимиты
- [ ] **98. Система репутации** — узел копит кредиты за отданные вычисления, тратит при использовании чужих
- [ ] **99. Режим «Поделиться мощностью»** — переключатель; узел регистрируется и принимает входящие задачи
- [ ] **100. Лимит ресурсов** — `systeminformation`; не более 20% GPU и 15% CPU; при превышении — пауза
- [ ] **101. Очередь входящих задач** — не более 3 одновременно; приоритет по балансу; таймаут 60 с
- [ ] **102. Маршрутизация запросов** — сервер ищет свободный узел с нужной моделью; иначе — локальная Ollama
- [ ] **103. Шифрование трафика** — TLS + end-to-end между узлами; владелец узла не видит чужих промптов
- [ ] **104. Согласие на участие** — явный экран при первом включении
